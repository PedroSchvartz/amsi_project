--------------------------------------------------------------------------------
  CONTRATO COM O BACKEND
--------------------------------------------------------------------------------

Fonte da verdade da API: openapi_ai.yaml (raiz do projeto)
Swagger local: /docs (backend rodando)

Fluxo de autenticação (implementado):
  1. POST /auth/token → recebe { access_token, primeiro_acesso }
  2. Token armazenado em localStorage
  3. Se receber 401 → handleResponse() em api.js chama logout() e redireciona para /

Fluxo de autenticação (PENDENTE — não implementado):
  - Se primeiro_acesso == true → redirecionar para tela de troca de senha
  - POST /auth/trocar-senha com token no header
  - Leitura do header X-Session-Expires a cada response autenticado
  - Timer de inatividade → POST /auth/logout + redirecionar para /

ATENÇÃO — logout atual:
  - logout() em auth.js faz localStorage.clear() mas NÃO chama POST /auth/logout
  - O token permanece válido no backend até expirar naturalmente
  - Corrigir: logout manual deve chamar logoutUser() da api.js antes de limpar localStorage

Regras de contrato (o backend sempre):
  - Aceita e retorna JSON
  - Nunca retorna o campo `senha`
  - Datas em ISO 8601 (ex: "2026-04-19T10:00:00")
  - Decimais como string no JSON

O frontend sempre:
  - Manda Authorization: Bearer <token> em todas as requisições autenticadas
  - Trata 401 → redireciona para login (já implementado em handleResponse)
  - Trata 403 → exibe mensagem de permissão insuficiente

ENUMs relevantes (vindos do banco):
  - natureza_enum: "Debito" / "Credito" (sem acento)
  - tipo_clifor_enum: "C" / "F" / "A"

--------------------------------------------------------------------------------
  CONVENÇÕES DO FRONTEND
--------------------------------------------------------------------------------

Nomenclatura:
  - Componentes: PascalCase (ex: LancamentoForm, ClienteList)
  - Pages: PascalCase + sufixo Page (ex: LancamentoPage)
  - Services: camelCase, agrupados por domínio em api.js
  - Estilos: arquivo CSS por componente, mesmo nome (ex: Lancamento.css)
  - Variáveis de domínio: português (seguindo o backend)

Padrões de estrutura:
  - Lógica de negócio e chamadas HTTP ficam nos services/
  - Componentes não chamam fetch diretamente — usam services/api.js
  - Pages são wrappers leves — não contêm lógica pesada
  - Rotas protegidas usam PrivateRoute ou AdminRoute

Antes de criar componente novo:
  1. Verificar se já existe algo aproveitável em components/
  2. Verificar se a rota da API já existe em openapi_ai.yaml
  3. Verificar se o serviço já está em services/api.js
