Leia o §6.3 do escopo primeiro para o contexto e as decisões de produto; este arquivo traduz aquilo em mudanças concretas por arquivo. **Não desvie das decisões já fechadas** — se algo parecer faltar, prefira perguntar a inventar.

> Stack: backend FastAPI + SQLAlchemy + PostgreSQL (`backend/`), frontend React + Vite (`AMSI_Frontend/src/`). Sem Alembic — migração manual via `backend/utils/bootstrap.py`. Testes: pytest em `backend/tests/`.

## Objetivo

Permitir criar **um lançamento idêntico para vários clifors de uma vez**, a partir do popup "Novo Lançamento". O usuário abre um modal-seletor de clifors (lista + filtros + checkbox), confirma a seleção, preenche os campos uma vez e cria tudo num clique. Lançamentos criados em lote compartilham um identificador `lote`.

**Regra de ouro:** o `POST /lancamento/` de criação única e seu contrato (`response_model=LancamentoResponse` de objeto único) **não mudam**. Tudo de massa é aditivo.

## Parte A — Backend

### A1. Modelo: nova coluna `lote`

**Arquivo:** [`backend/models/lancamento.py`](../../backend/models/lancamento.py)

`BigInteger` já está importado. Adicionar a coluna na classe `Lancamento` (sugestão: logo após `estorno`):

```python
lote = Column(BigInteger, nullable=True)
```

- **`BigInteger`, não `Integer`** — o valor é um epoch em **milissegundos** (~1,78 trilhão hoje), que estoura o `INT` de 32 bits do Postgres.
- `nullable=True` — **nulo = lançamento individual; preenchido = membro de um lote**.

### A2. Migração idempotente no bootstrap

**Arquivo:** [`backend/utils/bootstrap.py`](../../backend/utils/bootstrap.py)

Espelhar o padrão de `_migrar_acesso_enum` (checagem via `information_schema` + `db.execute(text(...))`, idempotente). Para um banco novo a coluna já nasce do `create_all`; para o banco de produção existente, é o `ALTER TABLE` que a adiciona.

```python
def _migrar_coluna_lote(db: Session):
    from sqlalchemy import text
    existe = db.execute(text(
        "SELECT 1 FROM information_schema.columns "
        "WHERE table_name='lancamento' AND column_name='lote'"
    )).first()
    if not existe:
        db.execute(text("ALTER TABLE lancamento ADD COLUMN lote BIGINT"))
        db.commit()
```

Registrar a chamada `_migrar_coluna_lote(db)` na **mesma sequência de migração** onde `_migrar_acesso_enum(db)` é invocado. O deploy já roda o `bootstrap` no Railway (ver [4.2](../escopo_futuro.md#42-bootstrap-como-comando-railway)).

**Arquivo:** [`backend/schemas/lancamento.py`](../../backend/schemas/lancamento.py)

**(b)** Novo schema de entrada do lote. ### A4. Endpoint + helper compartilhado

**Arquivo:** [`backend/routes/lancamento.py`](../../backend/routes/lancamento.py)

- Adicionar `from datetime import datetime` (hoje só importa `date`).
- Importar os novos schemas (`LancamentoMassaCreate`, `LancamentoMassaResponse`).

**Helper compartilhado** (monta a ORM **sem** validar nem commitar) — usado pelas duas rotas para não divergirem:

```python
def _montar_lancamento(dados: dict) -> Lancamento:
    return Lancamento(**dados)
```

Refatorar `criar_lancamento` para usar o helper (validação e `commit` continuam **idênticos** — só a construção do objeto passa pelo helper). **Não altere o comportamento observável da rota única.**

**Novo endpoint** (validação em lote → um único commit atômico → inadimplência por clifor distinto → relatório):

```python
@router.post("/massa", response_model=LancamentoMassaResponse)
def criar_lancamentos_massa(
    dados: LancamentoMassaCreate,
    db: Session = Depends(get_db),
    _=Depends(exige_operador_ou_admin),
):
    if not db.query(Usuario).filter(Usuario.id_usuario == dados.id_usuario_fk_lancamento).first():
        raise HTTPException(status_code=404, detail="Usuário não encontrado")
    if not db.query(tipo_conta).filter(tipo_conta.id_tipo_conta == dados.id_tipo_conta_fk).first():
        raise HTTPException(status_code=404, detail="Tipo de lançamento não encontrado")

    ids = list(dict.fromkeys(dados.ids_clifor))   # dedup preservando ordem
    if not ids:
        raise HTTPException(status_code=422, detail="Selecione ao menos um cliente/fornecedor")

    encontrados = {
        r.id_clifor for r in
        db.query(ClienteFornecedor.id_clifor).filter(ClienteFornecedor.id_clifor.in_(ids)).all()
    }
    faltantes = [i for i in ids if i not in encontrados]
    if faltantes:
        raise HTTPException(status_code=404, detail=f"Cliente/Fornecedor não encontrado: {faltantes}")

    lote = int(datetime.utcnow().timestamp() * 1000)   # ms, igual ao exp_ms do token

    novos = []
    for id_clifor in ids:
        l = _montar_lancamento({
            "id_usuario_fk_lancamento": dados.id_usuario_fk_lancamento,
            "id_clifor_relacionado_fk": id_clifor,
            "id_tipo_conta_fk": dados.id_tipo_conta_fk,
            "valor": dados.valor,
            "data_vencimento": dados.data_vencimento,
            "natureza_lancamento": dados.natureza_lancamento,
            "observacao": dados.observacao,
            "lote": lote,
        })
        db.add(l)
        novos.append(l)

    db.commit()                                  # atômico: 1 commit para todo o lote
    for l in novos:
        db.refresh(l)
    for id_clifor in set(ids):
        atualizar_inadimplente(id_clifor, db)

    return LancamentoMassaResponse(
        lote=lote,
        total_criados=len(novos),
        ids=[l.id_lancamento for l in novos],
    )
```

**Notas:**
- **Atomicidade:** um único `commit` no fim. Se qualquer insert falhar, a transação inteira reverte (nada é criado). Validar **tudo antes** de inserir qualquer coisa já evita a maioria dos erros parciais.
- **Log de atividade é automático:** o `ActivityLogMiddleware` ([`backend/utils/activity_log_middleware.py`](../../backend/utils/activity_log_middleware.py)) grava todo POST com status `< 300`. Não adicione log manual. (O `_parse_entidade` mapeia `/lancamento/massa` → entidade `lancamento`, `id_entidade=None` — ok.)
- **RBAC:** `exige_operador_ou_admin` — Consulta lê a lista no seletor (via `GET /cliente_fornecedor/`) mas **não** cria.
- **Ordem da rota:** declarar `@router.post("/massa")` antes de qualquer rota POST com path-param conflitante não é problema aqui (o único POST é `/`), mas mantenha `/massa` como literal.

### B3. Integração no `LancamentoModal`

**Arquivo:** [`AMSI_Frontend/src/components/LancamentoModal.jsx`](../../AMSI_Frontend/src/components/LancamentoModal.jsx)

- **Estado novo:** `clifforsSelecionados` (array de ids). Manter o `<select>` único atual como o caminho de 1 item; quando `clifforsSelecionados.length >= 1`, ele passa a ser a fonte de verdade.
- **Botão verde** na linha "Cliente / Fornecedor", canto direito (estilo do `.save`): abre o `MassaCliforSeletorModal`. Rótulo/ícone que comunique "selecionar vários" (ex.: ícone de pessoas + "Vários"). Quando já há 2+ selecionados, vira **"editar seleção"**.
- **Modo dinâmico por contagem:**
  - `length <= 1` → título **"Novo Lançamento"**, comportamento atual intacto (usa o `<select>` ou o único id selecionado).
  - `length >= 2` → título **"Novo Lançamento em Massa"**; substituir o `<select>` por um texto read-only **"N clientes/fornecedores selecionados"**.
- **Descrição:** permanece livre/limpa (sem prefixo "Lançamento em massa:").
- **No `handleSubmit`:**
  - **1 selecionado** → fluxo atual (`createLancamento`), inalterado.
  - **2+ selecionados** → abrir **modal de confirmação** com **contagem + valor total** (`N × valor`) e a lista de nomes **recolhível**. Ao confirmar, chamar `createLancamentoMassa({ ids_clifor, ...campos })`. A natureza é computada como hoje (tipo de conta + flip de reembolso).
  - Resultado: **um** toast só (ex.: "3 lançamentos criados") — integra com o rework de toasts da [2.8](../escopo_futuro.md). Nunca N toasts.

### B4. Exibição do `lote` na lista

**Arquivo:** [`AMSI_Frontend/src/pages/ListaLancamentosPage.jsx`](../../AMSI_Frontend/src/pages/ListaLancamentosPage.jsx)

A linha de status é renderizada por `statusLabel(l)` (def. ~linha 358) e usada no `<td>` da tabela (~linha 684). **Inline, na mesma célula de status**, após o badge, mostrar o chip de lote **quando `l.lote` não é nulo** — sem quebrar linha, sem coluna nova:

```javascript
const loteLabel = (lote) => {
    const d = new Date(lote);                       // ms → Date
    const dia = String(d.getDate()).padStart(2, '0');
    const mes = String(d.getMonth() + 1).padStart(2, '0');
    return `Lote ${dia}/${mes} #${String(lote).slice(-8)}`;  // ex.: "Lote 14/06 #00123456"
};
```

No `<td>` de status:
```jsx
<td>
  {statusLabel(l)}
  {l.lote != null && <span className="badge badge-lote" title={`Lote ${l.lote}`}>{loteLabel(l.lote)}</span>}
</td>
```

Adicionar a classe `.badge-lote` no CSS do componente (tom neutro, discreto, seguindo as `--vars` do tema). Opcional: repetir o chip nas views de detalhe que já usam `statusLabel` (linhas ~783, ~985, ~1145).
