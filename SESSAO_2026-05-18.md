## Planejamento — próxima sessão

### Tratamento de erro abrangente

Quatro frentes a implementar em conjunto:

#### 1. Error Boundary
- Criar `components/ErrorBoundary.jsx` (class component — hooks não capturam erros de render)
- Envolver o `<Layout />` (rotas protegidas) com o boundary
- Tela de fallback amigável: ícone, mensagem genérica, botão "Voltar ao início"
- Evita tela branca total quando um componente crasha

#### 2. Erros de API mais descritivos
- Em `handleResponse` (`services/api.js`), mapear status HTTP para mensagens em português:

| Status | Mensagem padrão |
|--------|----------------|
| 400 | Requisição inválida — verificar os dados enviados |
| 403 | Sem permissão para esta ação |
| 404 | Registro não encontrado |
| 409 | Conflito: registro já existe ou possui dependências |
| 422 | Dados inválidos: `{detalhe do backend}` |
| 500 | Erro interno do servidor. Tente novamente mais tarde. |

- Preservar a mensagem específica do backend quando ela vier em `detail` (já feito parcialmente)
- Não expor stack traces nem mensagens técnicas ao usuário

#### 3. Erros de rede / offline
- Adicionar listener `window.addEventListener('offline', ...)` no `MonitorSessao` (ou componente dedicado)
- Banner fixo no topo quando `navigator.onLine === false`: "Sem conexão — verifique sua internet"
- O retry de banco adormecido (`VITE_DB_SLEEP_MS`) já existe; unificar feedback com o banner
- Ao voltar online, esconder o banner e tentar recarregar a página atual automaticamente

#### 4. Padronizar feedback visual
- Problema atual: mistura de `setErro` (banner inline vermelho) e `mostrarToast` conforme a página
- Proposta: centralizar em um `ToastProvider` global (ou usar a lib `react-hot-toast` / `sonner`)
  - Erros → toast vermelho, duração 5 s
  - Sucesso → toast verde, duração 3 s
  - Avisos → toast amarelo, duração 4 s
- Manter banner inline apenas para erros de validação de formulário (associados a campos específicos)
- Remover `setErro` / estado local de erro das páginas que hoje usam para API errors
